# Feature Specification: Local-first PWA Photo Inventory

- **Feature ID**: `001-local-first-pwa-inventory`
- **Status**: Draft / Ready for planning
- **Source**: `deep-research-report.md`
- **Date**: 2026-06-22
- **Language**: zh-TW

## 1. Problem Statement

家庭或個人經常需要管理衣物、工具、收藏品、藥品、備品與雜物，但現有做法多半依賴記憶、相簿或試算表，難以快速回答「物品在哪裡」、「哪些物品快到期」、「上次移動到哪個櫃子」等問題。系統需要讓使用者用手機快速拍照建檔、指定位置、離線保存，並在需要時透過照片、關鍵字、位置或標籤找回物品。

## 2. Product Goal

建立一個照片導向、位置導向、local-first 的 PWA 收納管理系統，使單一使用者或小型家庭可以：

1. 快速新增物品照片與基本資料。
2. 以家庭式位置樹管理房間、櫃位、抽屜、掛勾與箱子。
3. 在離線狀態下仍可新增、搜尋、瀏覽與移動物品。
4. 網路恢復後將本地變更同步到雲端。
5. 保留物品移動與編輯歷史。
6. 支援 JSON/CSV 匯出、備份與還原。

## 3. Users and Personas

### 3.1 Primary User: 家庭管理者

- 管理家庭衣物、日用品、工具與備品。
- 主要使用手機拍照與查詢。
- 偶爾使用桌面整理大量資料。
- 最重視快速新增、搜尋與知道物品目前位置。

### 3.2 Secondary User: 個人收藏或租屋使用者

- 管理收藏品、搬家箱、出租屋物品或季節用品。
- 需要位置、標籤、照片與移動歷史。
- 可能需要匯出備份。

## 4. Scope

### 4.1 MVP In Scope

- PWA 可安裝與離線啟動。
- 物品 CRUD。
- 拍照或上傳照片。
- 圖片壓縮與縮圖。
- 位置樹 CRUD。
- 標籤與分類。
- 本地搜尋與過濾。
- 物品移動歷史。
- 到期日與 App 內提醒。
- IndexedDB 本地資料庫。
- Outbox/Inbox 基本同步。
- JSON/CSV 匯入匯出。
- 備份與還原的資料格式。

### 4.2 Out of Scope for MVP

- 多人即時協作。
- AI 自動分類。
- OCR。
- 條碼或 QR code 掃描。
- 複雜家庭平面圖。
- 全平台可靠推播。
- CRDT 協作同步。
- 原圖無限制雲端備份。

## 5. User Stories

### P0 Stories

1. 作為使用者，我想用手機拍照新增物品，並立即儲存在本機，讓我即使網路不好也能完成整理。
2. 作為使用者，我想把物品放到「房間 → 櫃位 → 抽屜/掛勾/箱子」的位置樹中，讓我之後能知道物品在哪裡。
3. 作為使用者，我想用名稱、標籤、分類、位置搜尋物品，讓我能快速找回需要的東西。
4. 作為使用者，我想在物品移動時留下紀錄，讓我知道上次把東西移到哪裡。
5. 作為使用者，我想離線瀏覽最近或全部本地物品，讓我在地下室、倉庫或網路不穩時也能使用。
6. 作為使用者，我想讓本地變更在網路恢復後自動同步，讓不同裝置可以取得最新資料。

### P1 Stories

1. 作為使用者，我想設定物品到期日或提醒日，讓我知道藥品、保健品或季節用品何時需要處理。
2. 作為使用者，我想匯出 JSON/CSV 與圖片資料，讓我可以備份或轉移資料。
3. 作為使用者，我想從備份還原資料，讓我在換裝置或資料遺失時恢復狀態。
4. 作為使用者，我想看到同步狀態與待同步數量，讓我知道資料是否已安全保存到雲端。

### P2 Stories

1. 作為家庭成員，我想分享某些位置或物品給其他成員，讓家庭可以共同維護。
2. 作為使用者，我想啟用系統通知，讓 App 可以在支援的平台提醒我。
3. 作為使用者，我想使用 AI 或 OCR 輔助分類與辨識，讓大量建檔更省力。

## 6. Functional Requirements

### 6.1 Item Management

- 系統必須支援建立、查看、編輯、刪除與還原物品。
- 每個物品至少包含名稱、分類、狀態、備註、目前位置、封面照片、建立時間、更新時間與版本。
- 刪除應採 soft delete，保留可還原期間。

### 6.2 Photo Capture and Upload

- 系統必須支援相機拍照。
- 系統必須支援檔案上傳作為 fallback。
- 系統必須在本地產生主圖與縮圖。
- 系統必須支援離線保存圖片 metadata 與可用縮圖。

### 6.3 Location Tree

- 系統必須支援樹狀位置。
- 位置節點必須支援 parent-child 關係。
- 使用者必須能依位置瀏覽該位置與子位置的物品。

### 6.4 Tags and Categories

- 系統必須支援標籤 CRUD。
- 系統必須支援物品與多個標籤關聯。
- 系統必須支援分類欄位。

### 6.5 Search and Filters

- 系統必須支援本地搜尋。
- 搜尋條件至少包含名稱、分類、標籤、位置、到期日與狀態。
- 離線時搜尋仍必須可用。

### 6.6 History

- 系統必須記錄建立、編輯、移動、照片新增與刪除事件。
- 移動事件必須記錄來源位置與目的位置。

### 6.7 Offline and Sync

- 所有 P0 操作必須先寫入 IndexedDB。
- 每個會影響雲端狀態的操作必須建立 SyncOp。
- 系統必須有 Outbox 追蹤待推送操作。
- 系統必須有 Inbox 或 change cursor 追蹤雲端增量。
- 衝突必須能被偵測，並至少提供 last-write-wins 或手動解決流程。

### 6.8 Import, Export, Backup

- 系統必須支援 JSON 匯出。
- 系統應支援 CSV 匯出。
- 系統應定義照片與 metadata 的備份 manifest。
- 還原流程必須先驗證 schema version。

## 7. Non-functional Requirements

### 7.1 Performance

- PWA 首次安裝後必須能離線啟動 App Shell。
- 搜尋與物品列表在中階手機上應維持可互動。
- 圖片壓縮不得長時間阻塞 UI；必要時應使用背景工作或分批處理。

### 7.2 Reliability

- 離線操作不得因同步失敗而遺失。
- 同步失敗必須保留於 Outbox 並可重試。
- 使用者必須能看見待同步狀態。

### 7.3 Privacy and Security

- 系統必須以 HTTPS 部署。
- 認證與授權必須保護每位使用者或家庭資料。
- 照片與物品資料不得被未授權使用者讀取。
- 可選支援敏感欄位本地加密。

### 7.4 Portability

- 使用者必須能匯出資料。
- 資料模型應保留 schema version，以支援未來 migration。

## 8. Acceptance Criteria

1. 使用者可以在離線狀態新增一筆含照片、名稱、位置與標籤的物品。
2. 使用者可以在離線狀態搜尋剛新增的物品。
3. 使用者可以移動物品到另一個位置，且歷史紀錄保留來源與目的位置。
4. App 重開後，離線資料仍存在。
5. 網路恢復後，待同步操作可以被推送並標示為已同步。
6. 相機權限被拒絕時，使用者仍可透過檔案上傳新增照片。
7. 使用者可以匯出包含物品、位置、標籤與歷史的 JSON。
8. PWA 可安裝，並可在首次成功載入後離線啟動基本 shell。

## 9. Open Questions

1. MVP 是否只支援單使用者，或需要家庭成員共享？
2. 圖片是否保存原圖，還是只保存壓縮主圖與縮圖？
3. 雲端首選供應商是否確定為 Supabase？
4. 是否需要第一版就支援通知權限與 Service Worker notification？
5. soft delete 保留期限是否固定為 30 天？
